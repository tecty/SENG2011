1. Poster
A poster is someone who want to post a task on PartyWhip and choose the tasker
they want.
1.1. [1]Posters can post one or more requests of food provision
1.1.1. [1]Posters must specify basic parameters for requests
1.1.1.1. [1]name and message of the request
1.1.1.2. [1]number of people
1.1.1.3. [1]location
1.1.1.4. [1]date and time
1.1.1.5. [1]food types
1.1.1.6. [2]the closing time of bids //couldn’t be later than event time
1.1.1.7. [1]budget for whole event
1.1.1.8. [1]status of the request // change by the progress (incl.
Processing, Deal, Finished, Cancelled)
1.1.2. [2]Posters can specify some other parametes
1.1.2.1. [2]A post can only select one parameter value for a field
(Couldn’t have multiple parameters with one field)
1.1.2.2. [2]Poster can see all the provided parameters
1.2. [1]Posters can view requests they posted // Even if it is finished
1.2.1. [1]Posters can view all requests they posted in a list // sorted by status
1.2.1.1. [3]Posters can sort the requests by the date
1.2.1.2. [2]Posters can filter requests by their status
1.2.1.3. [4]Posters can sort the requests by the latest bid
1.2.1.4. [4]Posters can filter requests by some other parameters
1.2.2. [1]Posters can choose bidders for an open request // change state
1.2.2.1. [2]bidders will be sorted based on their bidding price and
bidding time // sorting
1.2.2.2. [2]Poster can see all bidders’ comments written by previous
posters.
1.2.2.3. [3]Poster can view bidders’ information on bidders’ profile //
Only bidders who bid this post and the post is not in state
Finished or Cancelled
1.2.2.4. [2]When a request is overdue, it should be automatically
cancelled // changing the state
1.2.2.5. [3]Posters can evaluate a bidder after a request is completed
1.2.2.5.1. [3]Poster can only rate his post
1.2.2.5.2. [3]The rate should be one to five stars in intergers
1.2.2.5.3. [3]Posters can rate the bidder for this request
1.2.2.5.4. [4]Posters can write comments to this bidder
1.2.2.6. [3]Poster will see the recommend stars of all the bidders who
bid the post.
1.2.2.6.1. [3]The recommend starts is base on the rate of other
posters’ rate to this bidder
1.2.2.6.2. [3]The recommend starts is base on the parameters of
this posts.
1.2.2.6.3. [3]Appear 0 rate if these rules are not applied, such as
there’s no rate to this bidder, this bidder has done
nothing about the given parameter of this post.
1.2.3. [1]Posters can manage his posts
1.2.3.1. [1]Posters can cancel a processing or deal state request which
created by themselves // change state, need to verify
1.2.3.2. [2]Posters can edit details of a request created by themselves
and not in state Finished or Cancelled // change states of
bidders needs to verify
1.2.3.3. [2]Posters can regret for selecting a bidder 6 hours prior the
service requesting time
1.3. [1]Posters can register and log in
1.3.1. [3]Posters can edit and provide contact details
1.3.1.1. [3]Phone number
1.3.1.2. [3]Email
1.3.1.3. [3]Name
1.3.1.4. [3]Address
1.3.1.5. [4]Additional information
2. Parameter
Parameter is the key value pair help poster to specify extra nature of the event.
2.1. [1]Parameters can only be CRUD (Change Read Update Delete) by admin
2.1.1. [1]Parameters cannot be hard deleted // need to verify, implement by a
deleted state in Parameters
2.2. [2]Parameter is a key-value pair, including // created admin
2.2.1. [2]nature of event
2.2.2. [2]level of food quality
2.2.3. [2]special diet
2.2.4. [2]religious restriction
2.2.5. [2]provision of alcohol
2.2.6. [2]age profile
2.2.7. [2]kitchen size
2.2.8. [2]indoor/outdoor
2.2.9. [3]additional provision of crockery/glasses/cutlery/chairs
3. Bidder
Bidder is who want to bid a task and finish the post when he choose by the poster.
3.1. [1]Bidders can view a list of posted requests
3.1.1. [2]Default list is sorted by relevant rate // need to verify
3.1.2. [2]Bidders can sort requests
3.1.2.1. [2]sort by latest bidding price
3.1.2.2. [2]sort by latest bidding time
3.1.2.3. [2]sort by relevance
3.1.3. [2]Bidders can filter requests
3.1.3.1. [2]filter by food type
3.1.3.2. [2]filter by region
3.1.3.3. [2]filter by due time
3.1.3.4. [2]filter by other parameters
3.1.4. [4]Bidders can only see requests with food types they can provide
3.2. [1]Bidders can place bids on requests
3.2.1. [1]Bid need to provide the offer budget and leave some comments
3.2.2. [4]get notification when a bid updates
3.3. [1]Bidders can register and log in
3.3.1. [3]Bidder must have a profile
3.3.1.1. [3]Bidder can edit its own profile // only bidder and poster can
edit
3.3.1.2. [3]including contact details
3.3.1.3. [3]including the information of restaurant (if applicable)
3.3.1.4. [3]including food types they can provide
3.3.1.5. [3]including past photos of catering services they provided
3.3.1.6. [3]including their prefer time for service provision (help to
match the post more effectively)
3.3.2. [3]Bidders can view posters’ contact information when they have been
selected for the posters’ requests
3.3.2.1. [3]Only can vidw the detail at Deal state of the corresponding
post
4. Admin
Admin can access all the necessary data to manage the website and provide
customer service.
4.1. [3]Admin can log into a system different from bidder and poster
4.2. [3]Admin can delete the account of bidders and posters
4.3. [3]Admin can CRUD all Post and Bid
4.4. [3]Admin can CRUD all Parameters
